2026 年 4 月的某天晚上,我把自己写了三个月的 RTM 服务从 docker 里停掉了。代码还在,容器不跑。

这不是删库跑路的故事,是一个**理性决定**。过程里我被反复逼问同一个问题 —— "你写的是不是错的",以至于不得不把"代码质量"和"该不该上生产"掰成两件事来想清楚。

这篇文章记录这个过程。

<!-- more -->

## 它是什么

一个 Go 写的实时消息中间件。核心数据:

- **7500 行代码**,20+ 模块
- 协议**对标 Agora RTM**:Ping/Login/Peer/Channel/Join/Leave/Presence/AttrSet/Ack 一整套
- **完整做了** QoS(AtMostOnce / AtLeastOnce / ExactlyOnce)、集群发现、可靠性重传、TokenAuth、IP 白名单、消息签名
- 部署在一台 1.7G 内存的华为云上,和语音房共存

怎么来的?语音房业务(自己的项目)需要信令 / 广播消息,我不想直接依赖声网 RTM —— 太贵、不够定制、也想自己把这件事搞清楚。于是就写了。

写到第三个月的某个时间点,功能基本对齐 Agora RTM 的文档。我满足地记录了这件事,然后 —— **没有真正上线**。

## 半年后回来看

半年里这个服务一直在跑,但**活动连接数一直是 0**。

语音房的代码里有这样一段:

```go
rtmClient := rtm.NewClient(...)
go func() {
    if err := rtmClient.Connect(ctx); err != nil {
        log.Printf("[RTM] 连接失败，将使用本地消息")
    }
}()
```

当初写的时候想的是"容错",现在回头看 —— **我把自己的服务当成可选依赖**。连不上就用本地消息,完全不影响业务。跑了七周,累计进出流量只有 33 MB。

为了确认这不是偶然,我扫了一遍服务器上其他服务:XClaw、ai-service、auth-service —— 没有一个在调它。

一个完成度 95% 的 RTM,零流量。

## 代码其实不差

这里我得先替代码说句公道话。

前人在 3 月份做过一轮代码审查,修了 21 个 bug。那份记录在飞书上,每一个修复都挺扎实:`pool.Close()` 误关整个连接池、`broadcastToRoom` 先复制再迭代、`writePump` 加写超时、`readPump` 收到 Pong 刷新 deadline 等等。修完之后核心路径是能跑的。

最近我又自己扫了剩下没被审过的 7000 行,找到一堆新问题:

- `conn.go` 里 `SendToConnection` 跟 `RemoveConnection` 之间有 race,写已关闭 channel 会 panic
- `AddUserConn` 是经典的 Load-Append-Store 非原子操作,多端登录会丢连接
- `Accept` 里 MaxConnections 检查和计数器递增之间有空窗,限制等于摆设
- `ReadDeadline` 只设了一次,心跳超时形同虚设
- 服务端主动发的"Pong"发成了应用层二进制消息,不是 WebSocket 标准控制帧 —— 客户端的 `SetPongHandler` 根本收不到
- Redis 操作一路没错误处理,Redis 挂了连接状态就静默错了

每一个都不复杂,都能修。工作量大概一到两天。

**但问题来了**:花一到两天修完这些 bug 之后,依然是零流量。修 bug 和"这东西存不存在"之间没有任何关系。

## 真正的困境

我花了一段时间才说服自己:**代码质量和业务价值是两件事。**

代码层面:Go 并发惯用法、模块划分、协议对标、审查意识 —— 这些是 B+ 到 A- 的水准。单人三个月写 7500 行还能跑起来的人不多。

业务层面:

- 语音房真需要 RTM 能力?接 Agora RTM 一行配置,它已经在用 Agora RTC
- 其他业务谁来接?没排期,没需求,没人问
- 私有部署 / 信创 / 合规场景?对,但**我服务的是自己的副业项目,没有这种客户**
- 当学习/练手作品?对,**这是最真实的动机**

承认最后一条花了很久。不是因为它丢人,是因为**把学习动机和生产动机混着做**,会导致既不像生产项目那样收敛功能,也不像学习项目那样写得彻底。结果是做了个"不上不下"的东西 —— 按生产标准看不敢用,按学习标准看又没做到极致(比如没对着 Matrix / XMPP 协议写深度博客)。

## 选"不上生产"不是承认失败

想明白之后,我反而松了一口气。决定停掉 rtm 容器 —— **不删代码,只是从运行态退下来**。

为什么这是正确的:

**1. 上生产的门槛比我想象的高。** RTM 是用户可感知故障的服务(消息丢了、不到、重复都是故障),需要 24/7 可用,需要告警,需要容量规划,需要升级不抖动。我一个人业余时间做,做不到。做不到就不上,上了必然出事。

**2. 零流量修 bug 是负收益。** 没有真实负载就没有真实反馈,闭门造车再美也是猜的。不如保持当前状态,真有业务来了再对着真问题修。

**3. 自研派不等于挂生产。** 这件事我之前没分清楚。Redis 作者 antirez 写博客频率比写代码高,SQLite 作者从来不上 SaaS,Kafka 作者到领英之前也是"内部写了一个"。自研的价值很多时候在于**深入研究过一个领域**,不在于运营一个产品。

## 那它怎么继续活

我重新给这个项目定位:**它是我的一个研究型项目,不是公司的基础设施**。

具体动作:

- **从业务系统里拆出来**,不再和语音房绑在一起
- **推到独立仓库**,README 大改,第一行就写清楚:对标的是协议(Matrix/XMPP/MQTT),不是 Agora 产品;当前状态是 learning-focused,不推荐生产使用
- **把已知问题公开化** —— 飞书那份 code review、这篇博客里新发现的 bug,都写进 BUGS.md。不是遮盖,是展示我知道
- **挑几个技术点写博客**(这是第一篇,后面会写 WebSocket 连接管理的坑、QoS 在弱网下的妥协、Matrix 协议对比这几个)
- **明确"我不做什么"**:不做运营、不做高可用、不承诺兼容性、不接业务真流量

这样一份代码吃三遍:写的过程已经学了一遍,挂个人作品集是第二遍,如果哪天有真业务来,底子还在是第三遍。

**不做生产,不代表它没价值**。

## 对"自研"这件事的看法

写完这个 RTM 之后,我对"自研"的看法变得具体了。过去我模糊地觉得"能自研是好的",现在我能分开讲:

自研**当作学习**:永远对。写完一个复杂系统和读十本书不一样,脑子里会永久留下"分布式系统真实难点长什么样"的形状。这是纯收益。

自研**当作产品**:要看条件。你要有持续迭代的决心,要有真实用户,要承担所有 bug 和安全责任,要跟成熟开源 / 商业方案 PK。大多数人业余项目不具备这些条件 —— 包括我。

自研**当作职业资产**:这是最被低估的。一个你能从里到外讲 30 分钟的项目,比"用过 Kafka"强太多。面试官问一个细节你能讲出当时的具体权衡 —— 这是写在简历上抄不来的。

我这个 RTM,作为学习和职业资产是 A 级,作为产品是 D 级。之前的错误在于把这三个维度混着评估,搞得自己很纠结。

## 结尾

停掉 rtm 容器的那一刻,我以为我会失落。实际上没有 —— **代码没少一行,学的东西也没少一点**。改变的只是它在系统拓扑图上的位置。

写这篇的时候,我把它从"XClaw 的内部组件"拖到了"Hugo 的个人项目"那一栏。这个拖动,就是一次架构决策。

下一篇我写 WebSocket 连接管理里那 7 个典型坑 —— 用我自己 `conn.go` 里找到的真代码、真 bug、真修法。比抄博客好玩。

---

**这篇的结论其实很短**:代码好不等于该上线。学得深不等于要运营。承认一件东西是"作品"而不是"产品",是对它更大的尊重。
